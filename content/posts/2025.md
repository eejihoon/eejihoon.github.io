## 2025-10-31 목요일

초대 링크 시스템을 완성했다. invite_tokens 테이블을 nanoid로 만들고, 생성/공개 기능을 구현했다. 수락/거절 페이지를 전면 개편해서 Daum 우편번호 서비스를 붙였고, 상태별로 버튼이 다르게 나오고 readonly 모드도 추가했다. Validation을 AlertModal로 바꾸고 모바일 이미지 영역도 개선했다. 협찬 혜택/형태 섹션 레이아웃도 손봤고, 설계 문서도 docs에 작성했다.

MOODY BEAUTY 버티컬을 새로 추가했다. DB 스키마랑 Server Action을 구현하고, Beauty 페이지 메인 갤러리 애니메이션 깜빡이는 거 고쳤다. 공통 UI 컴포넌트도 분리해뒀다.

제안 페이지를 여러모로 개선했다. UI를 다듬고 필수 멘션 기능을 추가했다. 제안 메시지 템플릿을 개편해서 동적 날짜 치환 기능({{startDate}}, {{endDate}})을 넣었다. 미리보기 모달 너비를 확대하고 크리에이터명 표시도 개선했다. 메시지와 초대 링크 관리 방식을 정리하고, username 표시할 때 @ prefix를 제거했다.

그 외에 DM 발송 방식 설명 문구를 개선하고, 크리에이터 이용약관 페이지를 추가했다. OpenGraph 메타데이터 설정을 완료했고, Vercel Cron이 GET 메서드를 지원하도록 수정했다. PR #87, #91 리뷰에 대응해서 Hotfix를 했고, TypeScript 빌드 에러와 ESLint 오류를 수정했다.

## 2025-10-30 수요일

MOODY FASHION 랜딩 페이지를 만들었다. 패션 특화 페이지에 광고 게시물 MockUI를 구현하고, 사전 등록 시스템에 이메일 필드를 추가했다. 모바일 반응형도 조정했다.

메인 페이지에서 HERO 03 크리에이터 추천 랭킹 모바일 이미지 그리드를 개선하고, Navigation CTA 텍스트를 바꿨다.

제안 메시지 템플릿 시스템을 만들었다. AlertModal을 통합하고 동적 날짜 치환 기능을 넣었다({{startDate}}, {{endDate}}).

브랜드 설정 페이지를 구현했다. Instagram DM 설정 페이지를 만들고 세션 관리 기능을 추가했다. 타입 오류도 수정했다.

DM Queue에 Rate Limiting이랑 Quiet Hours를 적용했다. Hydration mismatch 문제를 근본적으로 해결했는데 서버 SSR 인증 상태 처리 방식을 바꿨다. Instagram 연동 알림을 개선하고 AlertModal에 줄바꿈 지원을 추가했다.

## 2025-10-29 화요일

Proposal 생성 로직을 리팩토링했다. createProposal과 createProposalWithDM을 통합하고, 공통 로직을 helpers로 추출했다. mentionRequirements 타입 불일치도 수정했다.

DM Queue를 개선했다. Rate Limiting이랑 Quiet Hours를 적용하고, DM 상태가 제안 취소 시 즉시 업데이트되도록 했다. 자동 취소 기능도 구현했다.

아키텍처를 여러모로 개선했다. Rate Limit 로직을 Next.js로 옮겨서 아키텍처를 단순화했다. Transaction deadlock 문제를 해결하고, 비동기 처리로 바꿔서 timeout을 방지했다. Express 미들웨어 순서를 수정해서 req.body undefined 문제도 고쳤다.

Hydration 문제를 해결했다. BrandUserContext 초기 상태를 수정하고, authLoading 중에는 기본값을 유지하도록 했다.

CLAUDE.md를 업데이트해서 기술 스택과 SOLID 원칙을 추가했다. playwright-server DB 결합도 문제점도 문서화했다.

디자인 프로토타입 페이지를 전체 제거하고, 하드코딩된 'MOODY' 브랜드명을 수정했다.

## 2025-10-28 월요일

Phase 2 DB 큐 워커 시스템을 구현했다. dm_queue 테이블을 만들고 상태 관리를 추가했다(pending/processing/sent/failed). HTTP Trigger 기반 워커를 만들고, Rate Limiting과 Quiet Hours를 적용했다.

DM 자동화 다이얼로그를 개선했다. username 매칭 오류를 수정하고, 다이얼로그가 닫히는 버그를 해결했다. 2명 전송 버그도 고치고 칩 검증 로직을 개선했다.

보안을 강화했다. JWT 서명 검증과 httpOnly 쿠키 원칙을 제대로 준수하도록 했다. JWT-DB 상태 불일치 문제도 해결해서 사용자명 마스킹 문제를 고쳤다.

DM 테스트 스크립트를 개선하고 선택자를 확장했다. TypeScript 빌드 에러를 수정하고, Cloud Run 테스트 가이드를 개선했다.

## 2025-10-27 일요일

Phase 1 로그인 안정성을 개선했다. CAPTCHA 완료를 DOM 기반으로 자동 감지하고, CAPTCHA와 2FA를 페이지 내용 기반으로 올바르게 구분하도록 했다. CAPTCHA 시 Context를 유지하고 스트리밍을 개선했다. 백업 코드 버튼 셀렉터도 여러 셀렉터를 시도하도록 개선했다.

로그인 완료 감지를 개선했다. 복합 DOM 요소 기반 감지로 바꾸고, loginCheckInterval을 DOM 기반으로 단순화했다. 재시도 로직과 에러 메시지도 개선했다.

화면 스트리밍을 개선했다. FHD 해상도를 지원하고, /api/streaming/connect에서 brandId 파싱을 수정했다.

보안을 강화했다. Next.js API Routes 프록시로 API 키 보안을 강화하고, GOOGLE_CLOUD_RUN_API_KEY를 Secret Manager에 추가했다.

간헐적으로 클릭이 차단되는 문제를 해결했다. 디버그 로그를 추가하고 정리했다.

Phase 1 Task 6-7에서 Instagram 2FA를 완전히 지원했다. 인증 코드 자동 입력을 구현하고, 백업 코드(8자리)도 지원해서 자동 입력되도록 했다. CAPTCHA 이후 2FA를 자동으로 감지하고, 에러 메시지 셀렉터를 수정하고 재시도 로직을 개선했다.

Instagram 연결 플로우 UX를 개선했다. 로그인 에러 시 폼을 유지하고, 무한 재시도를 방지하도록 플래그 방식으로 바꿨다. username 저장 문제도 해결했다.

OG parser 성능을 최적화했다. 병렬 요청으로 최대 70% 속도를 단축했고, User-Agent fallback으로 오늘의 집 403 에러를 해결했다. 오늘의 집은 특별 처리해서 LinkedIn Bot을 직접 사용하도록 했다.

Dockerfile 빌드를 수정하고 로컬 Docker 테스트를 성공했다. Cloud Run 배포 가이드를 작성했는데 이미지 푸시 방식과 Secret Manager 사용법을 정리했다.

PR#81 리뷰를 반영해서 keyboard.type 버그를 고치고 디버그 로깅을 개선했다.

## 2025-10-23 수요일

Phase 1에서 WebRTC를 WebSocket + CDP로 전환했다. ADR-001을 작성하고, CDP Screencast와 Socket.io 기반으로 화면 스트리밍을 구현했다. WebRTC 관련 코드를 정리하고 webrtc 디렉토리를 streaming으로 리네임했다. ScreenStreamingService 예외 처리와 타입도 개선했다.

DM 페이지 성능을 개선했다. ScreenStreamClient 메모리 누수를 방지하고, 네트워크 로깅을 추가했다.

인증과 접근 권한을 개선했다. 미로그인 사용자가 Preview에 접근할 수 있도록 허용하고, Anonymous 사용자에게는 크리에이터 총 개수를 비공개했다. /creators 페이지에서 무한 Pending 버그도 수정했다.

브랜드 사용자 비밀번호 재설정 기능을 구현했다. URL params가 즉시 반영되도록 하고 로그인 리디렉트를 개선했다. OG parser를 개선해서 중복 처리를 제대로 하고, 광고 모아보기 툴팁 버그를 수정했다. Vercel Web Analytics를 추가했다. 코드 리뷰를 반영해서 세션 관리, 에러 처리, 보안을 개선했다.

병렬 워커 스크립트를 개선했다. 재시도 옵션을 추가하고, 세션 자동 체크를 넣었다. 워커별 실행 시간과 통계 로깅도 추가했다.

## 2025-10-20 일요일

크리에이터 스크래핑 파이프라인 전체를 구현했다(Phase A-F).

Phase A에서는 환경 설정과 로깅 시스템을 구축했다(Winston).

Phase B에서는 오케스트레이터를 만들었다. MICRO-BATCH 아키텍처를 적용하고, Profile/Feed 분리 수집 전략을 사용했다. 병렬 워커를 지원하도록 userDataDir을 분리하고 Run ID를 공유했다. 워커별 수동 로그인 스크립트를 만들고 Lock 충돌을 해결했다.

Phase C에서는 Puppeteer 기반으로 Instagram GraphQL 수집을 구현했다. 캐러셀 이미지 전체를 지원하고, 조회수(view_count) 필드를 추가했다. 스키마 순환 의존성도 해결했다.

Phase D에서는 Supabase Storage 업로드를 구현했다. 중복 방지 로직을 넣어서 Storage 비용을 95% 절감했다.

Phase E에서는 Drizzle ORM 기반 UPSERT 로직으로 DB에 저장했다. 프로필 이미지는 매번 갱신되도록 했다.

Phase F에서는 Gemini AI로 이미지 분석을 구현했다. 미적 점수, 색감 분석, 태그 자동 생성을 넣었다.

프로덕션 스크립트로 scrape-full.ts와 scrape-parallel.sh를 구현했다. 자동 재시도와 통계 로깅을 추가했다.

환경변수로 GEMINI_API_KEY와 GEMINI_MODEL을 통합하고, 파이프라인 구현 요약 문서와 사용법 가이드를 작성했다.

* **관찰:** "엉망인 코드베이스로도 어떤 사람들은 가치를 만들어낸다"는 사실을 인지함. 이는 기술적 완벽성보다 시장 가치 증명이 우선될 수 있다는 점을 이해하고 있음을 시사.
* **분석:** 이 명제는 '기술 부채(technical debt)'의 개념과 직결됨. 가치를 빠르게 만들기 위해 부채를 지는 것은 유효한 전략. 그러나 이 전략의 리스크는 '부채 상환 시점'을 놓치는 것. 현재의 '엉망인 코드'가 미래의 개발 속도를 심각하게 저해하여 가치 창출 능력을 떨어뜨리는 변곡점이 반드시 존재함.
* **제언:** "가치를 만든다"는 상태를 넘어, **"가치를 지속적으로, 예측 가능하게 만드는가?"**로 질문을 확장할 필요가 있음. 현재의 코드베이스가 새로운 기능 추가나 수정에 드는 시간을 어떻게 변화시키고 있는지, 그 속도의 변화를 의식적으로 관찰하고 기록해두는 것이 좋음. 이것이 기술 부채 상환 계획을 세울 객관적 근거가 됨.

3. '중요한 것'에 대한 다양한 관점에 관하여

* **관찰:** 'CS 기본기'를 중시하는 관점과 '비즈니스 가치'를 중시하는 현재 자신의 관점이 충돌한다고 느낌.
* **분석:** 이는 우선순위의 차이일 뿐, 대립하는 가치가 아님. 두 관점은 서로 다른 종류의 리스크를 관리함.
    * **CS 기본기:** 예측 불가능하고 디버깅이 어려운 시스템 레벨의 **'기술 리스크'**를 방지. 당장 필요 없지만, 문제가 터지면 근간을 흔듦.
    * **비즈니스 가치:** 제품이 시장에서 외면받을 **'시장 리스크'**를 방지. 항상 시급하며, 생존에 직결됨.
* **결론:** 현재 단계에서 '시장 리스크'를 최우선으로 관리하는 것은 합리적인 판단. 다만 '기술 리스크'를 완전히 무시하는 것이 아니라, "감당 가능한 수준으로 관리하고 있다"는 인지가 필요. 지금은 그 리스크를 감수하는 단계일 뿐.


종합

일기 전반에 걸쳐 '상황적 맥락(context)'의 중요성을 체득하는 과정이 나타남. 외부의 절대적인 기준들에 의문을 제기하고, 현재 자신의 좌표(프로덕트 단계, 팀 상황)에 근거하여 의사결정 기준을 재정립하고 있음. 이는 독립적인 개발자 또는 리더로 성장하는 과정에서 나타나는 긍정적인 패턴.
